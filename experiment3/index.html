Experiment 03 - Alternate Worlds



Description

In this project, we use a generative context-based map creator that stores map values in the form of a string-grid. By taking into account what tiles were placed next to each other, we were able to add tile interactions like walls.

For dungeon generation, the map is also populated by a number of extra chambers, which are then connected to the central chamber using a simple pathing algorythem. For the overworld map, all features are generated based on a simple noise function that acts like a heightmap for the final product.

Technical

The maping model used allows the user to directly edit the map that had been gererated, allowing for real-time feedback. It should be noted that, due to the way that the program checks for wall geometry, editing in non-standard floor plans may result in a buggy final render.

The adjacency algorhythm is rather simple, wherein it takes the four ajacent tiles, and encodes whether they are the target tile into the output using a four-digit binary number. This allows for the wall, shore-line, and loot generation to act the way that it does.

The pathing algorhythm that is used to ensure all dungeon rooms are connected is super basic, simply traveling in two straight lines to go from the new room's geometric center to that of the main hall. If you're curious, you can see any generated paths as denoted by a '+' on the map.

My only failing is that, while it does work as is, there are still some incongruencies between the text map and the rendered map due to me checking for chests and gems where walls should be.

Phineas Asmelash: Learning about bitmasking was a fun experience in this project. I think I did pretty well with this project. Especially with the variety of tilesets I integrated into the scene. Playing around with those different textures was nice, but I think I could have been more methodical with how I approached indexing the tilemap. It took me a while to figure out that I can just put a grid overlay on the tilemap to find the indexing of each tile element in the tile map, so that was some wasted time being inefficient. In the future, I would like to improve two things about this project. The first is to abstract the math away from the codebase by decoupling the long, mathy lines of code and putting them in their own function. This would especially help with the readability of the generateGrid() function. The second one is to come up with a more cohesive set of tilesets to create an overworld tile map to further improve the aesthetics of the project.
